# Phase 1 — The Fork Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce patched `asusd` / `asusctl` / `supergfxd` binaries in which every shipping v0.1 feature (thermal, GPU mode, keyboard backlight, charge threshold, fan curves, safe AC rule, physical mux) works correctly on ASUS TUF Gaming A15 FA507NV. Output: our fork tree, a maintained patch series, and a Phase 1 exit report proving parity on the four-state hardware matrix.

**Architecture:** Independent fork model — not downstream. Task 1-2 build 6.3.8 alongside 1.0.1 and pick the fork base from real hardware data. Task 3 sets up `fork/{asusctl,supergfxctl}/` as gitignored working trees plus `patches/` as the source-of-truth patch series (Debian quilt-compatible). Tasks 4-7 land one patch per feature on its own feature branch, one PR each, user merges. Task 8 is the AsusMuxDgpu physical-mux hardware test. Task 9 is the four-state regression matrix. Task 10 is the exit report.

**Tech Stack:** Rust (rustup, cargo, rustc ≥ 1.82), Linux 6.8 with `asus-nb-wmi`, systemd, dbus, `hwmon`, `gh` CLI for PRs, FA507NV hardware. Ubuntu 22.04 Jammy host.

## Global Constraints

- File length hard limit: 300 lines (CLAUDE.md rule 1) — **applies to code and shipping artifacts, not to this plan or verification reports.**
- One responsibility per file (CLAUDE.md rule 2) — applies to any new binaries we write (e.g. Task 6 Path B).
- Real hardware only — no VMs, no emulators (CLAUDE.md testing rule).
- Existing FA507NV state must be preserved unless a task explicitly migrates it: `/etc/modprobe.d/nvidia-custom.conf`, `/etc/systemd/system/battery-charge-threshold.service`, kernel cmdline. Do not modify.
- Independent fork model (memory: `feedback_independent_fork.md`). No "defer to v0.2 pending upstream fix" language. Broken features get fixed in our tree for v0.1.
- PR-and-merge workflow (memory: `feedback_workflow.md`). Every patch task ends with `gh pr create`. Never `gh pr merge`, never merge to `main` locally.
- Four-state test matrix per feature: AC+idle, AC+load, battery+idle, battery+load. Tasks 4-8 test their patch in ≥ 2 states; Task 9 covers the full matrix.
- `99-nvidia-ac.rules` from upstream supergfxctl must NEVER be installed live on FA507NV (crash trigger — Phase 0 report).
- Upstream trees stay in `upstream/` (gitignored, unchanged). Our fork tree lives in `fork/` (gitignored). Only `patches/`, `scripts/`, `docs/`, and any new source we author (e.g. `src/asusctl-fanctl/` if Path B) are committed.
- Fork base decision (default assumption for this plan: **1.0.1**) is finalised in Task 2. If Task 2 picks 6.3.8, Task 6 collapses to a verify-only step and Tasks 4-5 patches target the 6.3.8 tree instead. Every patch task's file paths listed below match the 1.0.1 layout observed in `upstream/asusctl/` — re-locate for 6.3.8 if base changes.

### Task 3: Fork tree and patches series

**Files:** Create `fork/asusctl/` and `fork/supergfxctl/` (gitignored). Create `patches/asusctl/series` (empty), `patches/supergfxctl/series` (empty). Create `scripts/build-fork-asusctl.sh` and `scripts/build-fork-supergfxctl.sh`. Modify `.gitignore` to add `fork/`.

**Consumes:** Task 2's chosen base tag.

**Produces:** A reproducible build path: `scripts/build-fork-asusctl.sh` clones upstream at the chosen tag into `fork/asusctl/` (if not present), applies every patch in `patches/asusctl/series` via `git am`, then `cargo build --release`. Same for supergfxctl. With empty series, both scripts produce vanilla-upstream binaries.

- [ ] **Step 1: Write `scripts/build-fork-asusctl.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail
BASE_TAG="${ASUSCTL_BASE_TAG:-v1.0.1}"   # override with env if Task 2 picked 6.3.8
UPSTREAM_URL="https://github.com/OpenGamingCollective/asusctl.git"
DEST="fork/asusctl"
SERIES="patches/asusctl/series"

mkdir -p fork patches/asusctl
touch "$SERIES"

if [ ! -d "$DEST/.git" ]; then
    git clone "$UPSTREAM_URL" "$DEST"
    git -C "$DEST" checkout "tags/$BASE_TAG" -b "fork/base"
fi

# Reset to pristine base then reapply series
git -C "$DEST" checkout fork/base
git -C "$DEST" branch -D fork/applied 2>/dev/null || true
git -C "$DEST" checkout -b fork/applied
while IFS= read -r p; do
    [ -z "$p" ] && continue
    git -C "$DEST" am "../../patches/asusctl/$p"
done < "$SERIES"

source "$HOME/.cargo/env"
(cd "$DEST" && cargo build --release)
```

### Task 7: Safe replacement for `99-nvidia-ac.rules`

**Files:** Create `patches/supergfxctl/0001-safe-nvidia-ac-rule.patch` OR create a design note in `docs/superpowers/specs/2026-07-01-nvidia-ac-rule-decision.md` if the safe rule ships as a Phase-2 packaging decision. Modify `patches/supergfxctl/series` if patching source.

**Consumes:** Phase 0 findings on the crash pattern.

**Produces:** A safe supergfxd behaviour on AC-state transitions. Never triggers `nv_acpi_powersource_hotplug_event` on FA507NV under any AC plug/unplug event.

- [ ] **Step 1: Choose safe-behaviour design**

Options (pick one and document why in the commit body):
- **7a — never start/stop nvidia-powerd on AC transitions.** Ship rule as `.disabled` in-tree. Nvidia-powerd only starts once at login via a supergfxd-managed systemd unit. FA507NV-safe by construction.
- **7b — stop-only on battery, no restart on AC.** Removes the AC-connect ACPI event that deadlocks. Nvidia-powerd restarts on next login.
- **7c — user opt-in.** Ship as `.disabled`; ship a `sudo asusctl-enable-ac-rule` helper that arms the upstream rule. Users with tested-safe firmware opt in.

Default recommendation: **7a**. Simplest, safest, no user footgun.

- [ ] **Step 2: Implement**

If patching supergfxctl source: modify the udev-rule generator or the daemon's AC-event handler to no-op the AC-connect start. Diff stays small.
If keeping as packaging-layer decision: no source patch; instead, write the design note and defer wiring to Phase 2 for `debian/`.

- [ ] **Step 3: Verify — the acid test**
