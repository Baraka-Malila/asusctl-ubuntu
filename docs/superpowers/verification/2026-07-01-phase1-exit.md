## What we shipped

### Fork base

- **asusctl**: OpenGamingCollective `6.3.8`. Chosen after side-by-side comparison with `v1.0.1` on FA507NV (Task 2). Every v1.0.1 bug on this hardware is *already fixed* upstream in 6.3.8: kbd backlight mpsc, charge threshold BAT probe, fan-curve CLI restoration. That collapsed the planned Phase 1 patch load from ~5 to what we actually did: 1 fix, plus one preemptive safety patch.
- **supergfxctl**: archived GitLab `5.2.7` (OGC mirror is empty). Vendored inside asusd 6.3.8's Cargo.lock too, so the build proves the dependency chain works.
- **Dbus name change inherited from 6.3.8**: `xyz.ljones.Asusd` (not `org.asuslinux.Daemon` of the v1.0.1 line). Documented as a Phase 2 packaging concern (ecosystem compat shim recommended, not required).

### Patches landed in the fork's `patches/` series

| # | Patch file | Purpose |
|---|---|---|
| asusctl/0001 | `power-source-sysfs-watcher.patch` | Unblocks asusd's per-power-source profile automation on Ubuntu 22.04. Upstream depends on logind's `OnExternalPower` dbus property, which was added in systemd 250; Ubuntu 22.04 ships systemd 249 where the property never updates at runtime. Our patch spawns a second tokio task in `CtrlPlatform::create_tasks` that polls `AsusPower::get_online()` (kernel sysfs, ground truth) every 2 seconds and fires the same policy chain. Additive — on 250+ both watchers may fire; daemon operations are idempotent. See PR #7 for full diagnosis + patch. |
| supergfxctl/0001 | `drop-99-nvidia-ac-rules.patch` | Removes `data/99-nvidia-ac.rules` from the supergfxctl source tree. Upstream's own Makefile doesn't install it (CHANGELOG marks it OPTIONAL), but any downstream packager who globs `data/*.rules` would sweep it in — and on FA507NV that rule is the trigger for the `nv_acpi_powersource_hotplug_event` LOCKUP. Removing at source-tree level makes accidental packaging inclusion impossible. See PR #4. |

Reproducible builds via:

```
./scripts/build-fork-asusctl.sh       # ASUSCTL_BASE_TAG defaults to 6.3.8
./scripts/build-fork-supergfxctl.sh   # SUPERGFXCTL_BASE_TAG defaults to 5.2.7
```

Each script clones upstream, checks out the base tag, applies every entry in the corresponding `patches/*/series` via `git am`, then `cargo build --release`. Empty series produces upstream-equivalent binaries; each patch is verifiable in isolation.

## What we honestly deferred, and to where

### Rescoped after Task 2 findings (base = 6.3.8 pre-fixed them)

- Original Task 4 (kbd backlight mpsc patch) — cancelled; 6.3.8 already fixed.
- Original Task 5 (charge threshold BAT probe patch) — cancelled as source patch; **rescoped to Phase 2 `debian/asusd.preinst`** that stops+disables the pre-existing `battery-charge-threshold.service` on install and re-enables on remove (asusd is the new sysfs owner). Actual Rust code needs no change.
- Original Task 6 (fan-curve CLI restore, Path A or B) — cancelled; 6.3.8's `fan-curve` subcommand works fully.

### Deferred to Phase 2 (packaging)

- **`asus-armoury-dkms` package.** The whole Armoury-Crate-equivalent surface (panel overdrive, PPT limits, GPU TGP, Dynamic Boost, refresh rate lock, boot logo) needs the `asus-armoury` kernel driver. In-tree since kernel 6.11; not in Ubuntu 22.04's stock 6.8. Design spec §5 already plans a DKMS backport package pinned to a `uejji/asus-armoury` commit. Zero kernel-upgrade required for the user.
- **`asus-backlight-fix` blacklist packaging.** Per the design spec — detect FA507NV-family and ship `/etc/modprobe.d/…` + `update-initramfs -u` postinst.
- **`battery-charge-threshold.service` migration in `debian/asusd.preinst`.** Replaces the old rescoped Task 5.
- **AsusMuxDgpu-mode backlight compatibility.** Open decision (spec option A vs B) — supergfxctl-hooked toggle vs documented manual revert.
- **`oneshot` side-effect (base_charge_control_end_threshold mutates)** — Phase 2 either shims via preinst snapshot/restore or patches upstream. Small.
- **Dbus rename compat** — decide whether Phase 2 packaging ships a dbus alias / policy allowing both `xyz.ljones.Asusd` and `org.asuslinux.Daemon` names for ecosystem-tool compat, or accepts the rename as the new default.

### Deferred to Phase 2.5 (feature-catalog work before Phase 3 GUI)

Introduced during Task 9.5 discussion: before building the GTK4+libadwaita GUI, we need an explicit **Armoury-Crate-feature catalog** mapping each Windows-side capability to (a) hardware presence on TUF/ROG, (b) Linux stack availability (daemon-native, kernel-side, external tool), (c) feature-parity ownership (asusd patch we ship, or third-party like GameMode). Without this catalog Phase 3 GUI becomes either underspecified or a mechanical clone of Slint's `rog-control-center` layout. Slint clone rejected by design spec §8.

Concrete Phase 2.5 items already identified:

- **`kbd_brightness_on_battery` field in asusd config.** ~30-line Rust patch, same shape as Task 9.5's sysfs watcher — wires kbd LED brightness switching into the AC-transition callback.
- **GPU mode per power source.** Same pattern. supergfxctl integration would auto-mode-switch on AC events.
- **Fan curve per (profile × power-source).** asusd already has `ac_profile_tunings` / `dc_profile_tunings` fields defaulting to `enabled: false`. Enable + populate.
- **Feature catalog document** — the artifact.

### Deferred to Phase 3 (GUI)

- **Native GTK4+libadwaita app** matching GNOME Settings / GNOME Software design bar. Not `rog-control-center` (Slint, non-native).
- **Per-app profile boost via `feralinteractive/gamemode` dbus signal.** Windows-equivalent "Armoury Crate game auto-detect" done the Linux way — user-authenticated, unprivileged processes can't drive thermal max without opting in via `libgamemode`.
- **Wayland-on-NVIDIA opt-in recipe** for `AsusMuxDgpu` users (delete `/etc/udev/rules.d/61-gdm.rules`, set `WaylandEnable=true` in `/etc/gdm3/custom.conf`), exposed as an experimental toggle in the display pane.

### Deferred to Phase 4 (distro expansion)

- Ubuntu 24.04 HWE testing on same feature matrix (systemd 255 — our sysfs watcher becomes double-poll idempotent-no-op; validates additive design).
- Ubuntu 26.04 Resolute + Debian bookworm/trixie support.
